# Architecture technique — Système de pointage JRV

## Contraintes structurantes

### 1. Agenda Google Calendar (source des séances)

L'agenda de l'association est un **agenda Google public**. Il constitue la seule source de vérité pour la liste des séances — aucune saisie dans l'application.

**Mode d'accès retenu : flux iCal public**

Tout agenda Google public expose un flux iCal (`.ics`) accessible sans clé API ni authentification :

```
https://calendar.google.com/calendar/ical/<calendar-id>/public/basic.ics
```

Le backend PHP récupère ce flux périodiquement, le parse et en extrait les événements (titre, date de début, date de fin). Aucun compte Google, aucun projet GCP, aucune clé API n'est nécessaire.

> Alternative écartée : l'API Google Calendar v3 (nécessite une clé API à gérer et un projet GCP — complexité inutile pour un agenda déjà public).

**Données extraites par événement**

| Champ iCal | Usage dans l'application |
|---|---|
| `DTSTART` | Détection de la séance en cours ou à venir |
| `DTEND` | Fin de la plage de la séance |
| `SUMMARY` | Titre affiché au participant au moment du pointage |
| `UID` | Identifiant stable de la séance, utilisé comme clé en base |

**Stratégie de mise à jour**

Le flux iCal est mis en cache côté serveur (fichier ou base de données) avec une durée de validité courte (ex. 15 minutes). Cela évite d'interroger Google à chaque requête participant tout en restant quasi temps réel.

### 2. Page web ultra-légère

La page participant doit fonctionner sur des téléphones anciens, avec une connexion parfois dégradée, sans installation.

**Principes**

- **Zéro framework JavaScript** : HTML, CSS et JS natifs uniquement. Pas de React, Vue, ni bundler.
- **Zéro dépendance externe** : pas de CDN, pas de polices distantes. Tout est servi depuis le même hébergement.
- **Une seule page** (`index.html`) avec un minimum de JS inline ou dans un fichier adjacent.
- **Mobile-first** : mise en page en colonne, boutons larges, tap-friendly.
- **Poids cible** : page complète < 50 Ko (HTML + CSS + JS).

**Interactions côté client**

1. Au chargement, une requête fetch récupère les séances depuis l'API PHP.
2. La séance en cours (ou la prochaine) est pré-sélectionnée.
3. L'autocomplétion du nom est gérée avec un `<datalist>` HTML natif — pas de bibliothèque tierce.
4. La soumission du pointage est un appel fetch (JSON) vers l'API PHP.
5. Un retour visuel (message de confirmation ou d'erreur) s'affiche sans rechargement de page.

**Accessibilité**

- Fonctionne sans JavaScript désactivé pour la partie formulaire (fallback `<form>` standard).
- Contrastes et tailles de police conformes aux recommandations WCAG AA.

### 3. Backend PHP sur hébergement mutualisé OVH

**Environnement**

L'hébergement mutualisé OVH impose les contraintes suivantes :

- Exécution **PHP uniquement** (pas de Node.js, Python, Ruby ni processus persistant).
- Version PHP configurable via le fichier `.ovhconfig` (PHP 8.1 ou 8.2 recommandé).
- Accès fichiers via **FTP/SFTP** ou SSH (selon l'offre).
- Pas de `exec()` ou de cron système — les tâches planifiées passent par le **gestionnaire de tâches OVH** (cron HTTP).
- Pas de contrôle sur les en-têtes serveur Apache au-delà du `.htaccess`.

**Base de données : SQLite (fichier local)**

L'offre mutualisée OVH ne fournit pas de base MySQL gratuite. Pour ce projet, **SQLite** est la solution retenue : la base est un simple fichier stocké sur le système de fichiers OVH, sans service externe ni credentials à gérer.

Justification par le volume de données :

| Indicateur | Valeur estimée |
|---|---|
| Participants par séance | ~50 |
| Séances par semaine | 2 |
| Pointages par an | ~5 200 |
| Pointages sur 5 ans | ~26 000 |
| Taille estimée du fichier `.db` | < 5 Mo |

PHP supporte SQLite nativement via `PDO_SQLite` (activé sur OVH). Le mode **WAL** (Write-Ahead Logging) est activé à l'initialisation pour gérer les écritures simultanées en début de séance sans blocage.

Le fichier de base de données est placé hors de la racine web (ou protégé par `.htaccess`) et peut être sauvegardé par simple copie FTP.

**Structure du backend**

```
/
├── index.html          ← page participant (frontend)
├── api/
│   ├── seances.php     ← GET  : liste des séances issues du flux iCal
│   ├── pointer.php     ← POST : enregistrement d'un pointage
│   └── admin/
│       ├── presences.php   ← GET  : liste des présences par séance (auth requise)
│       └── supprimer.php   ← POST : suppression d'une entrée erronée (auth requise)
├── cache/
│   └── agenda.ics.cache    ← cache local du flux iCal
└── .ovhconfig              ← version PHP, environnement
```

**Schéma de la base de données**

Deux tables suffisent :

```sql
-- Known members and ad-hoc visitors
CREATE TABLE IF NOT EXISTS attendees (
    id          TEXT PRIMARY KEY,   -- UUID v4, generated by PHP
    nickname    TEXT NOT NULL UNIQUE
);

-- Attendance records
CREATE TABLE IF NOT EXISTS checkins (
    id              TEXT PRIMARY KEY,               -- UUID v4
    session_uid     TEXT NOT NULL,                  -- iCal event UID
    attendee_id     TEXT NOT NULL REFERENCES attendees(id),
    checked_in_by   TEXT REFERENCES attendees(id),  -- NULL = self check-in
    created_at      TIMESTAMP NOT NULL
);

PRAGMA journal_mode=WAL;  -- allow concurrent writes without blocking
```

> UUIDs are stored as `TEXT(36)` and generated in PHP via `sprintf('%s-%s-%s-%s-%s', ...)` from `random_bytes()`. SQLite has no native UUID type.

**Sécurité**

- Les routes `admin/` vérifient un token statique transmis en en-tête HTTP (`Authorization: Bearer <token>`). Pas de session PHP — sans état.
- Toutes les entrées utilisateur sont validées et échappées avant insertion (PDO avec requêtes préparées).
- Le répertoire `cache/` est protégé par un `.htaccess` interdisant l'accès direct.
- L'API répond en JSON avec les en-têtes CORS restreints au domaine de la page.

**Cron OVH — mise à jour du cache iCal**

Un cron HTTP OVH appelle `api/refresh-cache.php` toutes les 15 minutes pour recharger le flux iCal en dehors des requêtes participant. En cas d'échec de rafraîchissement, le cache précédent est conservé.

## Points d'attention

- **Pas de HTTPS forcé par défaut sur OVH mutualisé** : à activer manuellement via `.htaccess` (`RewriteRule` vers `https://`) et le certificat SSL Let's Encrypt fourni par OVH.
- **Limite de mémoire et temps d'exécution PHP** : le parsing iCal doit rester rapide ; limiter le nombre d'événements récupérés (ex. 30 derniers jours + 90 jours à venir).
- **Pas de WebSocket** : les mises à jour en temps réel ne sont pas possibles sur hébergement mutualisé. L'interface responsable se rafraîchit par polling simple (ex. toutes les 30 secondes).
